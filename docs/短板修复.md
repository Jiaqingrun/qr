# QR 本地知识库 · 短板修复计划

> **用途**：汇总 QR 在架构、产品、治理、检索、Cursor 集成等方面的**已知短板**，按模块给出分阶段修复/升级路线与验收标准。  
> **何时读**：规划季度迭代、写 `EVOLUTION_PLAN` 新项、或 `qr evolution sync` 前对照优先级。  
> **互补文档**：[`EVOLUTION_PLAN.md`](./EVOLUTION_PLAN.md)（产品向跟踪）· [`RETRIEVAL_UPGRADE_PLAN.md`](./RETRIEVAL_UPGRADE_PLAN.md)（检索专项）· [`AI_SKILL_ASSESSMENT.md`](./AI_SKILL_ASSESSMENT.md)（评测口径）· `standards/STANDARDS.md` §四（设计者 / AI 实现 / 验收）  
> **状态**：2026-06-28 起草 · 各条默认 **待办**；完成后在本表与 `EVOLUTION_PLAN` 同步勾选。

## 0. 使用说明

### 0.1 短板分类

| 类型 | 标记 | 含义 |
|------|------|------|
| **刻意不做** | `维持` | 当前版本不修复，仅记录边界与触发条件 |
| **结构性取舍** | `取舍` | 单机/离线定位下的已知代价，可做缓解不做根治 |
| **设计完整、执行不足** | `闭环` | 机制已有，需产品化或自动化 |
| **真缺口** | `缺口` | 目标变大时会卡脖子，应排期修复 |

### 0.2 优先级

| 级别 | 含义 |
|------|------|
| **P0** | 直接影响设计者验收与资产沉淀，建议 1～4 周内启动 |
| **P1** | 影响可靠性或误判风险，建议 1～2 月内 |
| **P2** | 平台边界与维护债，按触发条件或季度排期 |
| **P3** | 与「刻意不做」相关，仅当 §8 触发条件满足 |

### 0.3 改完自检（通用）

```bash
conda activate qr && pip install -e ~/QR/dev/qr
python3 -m unittest discover -s tests
qr doctor
qr eval rag
qr evolution sync --full
```

## 2. M1 · 设计者体验与验收

> **短板**：规范已定义「设计者 / AI 实现 / 验收」，Web/CLI 仍偏工程师；非编程背景设计者难以独立完成闭环。  
> **类型**：`闭环` + `缺口`

### M1-1 · Web「设计者验收」向导 【P0 · 已完成 2026-06-28】

**目标**：在 Web 洞察页或项目页提供固定三步，对应 `standards` §四验收清单。

**内容**：
- 按钮/卡片：**① 运行 doctor ② 点验功能 ③ 写决策**
- doctor 结果摘要展示（成功绿 / warn 黄 / 需处理红）
- 一键跳转 `qr log --type decision` 表单（Web 已有 note API 则复用）
- 文案用**非程序员**表述（「检查系统是否正常」「试用刚改的功能」「记下为什么这样选」）

**改动范围**：`qr/web.py`（API 若需）、`qr/static/` 洞察或项目视图、`docs/USE_CASES.md` 增补一节

**验收**：
- Web 上完成三步后，时间线出现对应 note/decision
- `qr doctor` 无需手打命令即可从 Web 触发并看结果
- 规范 §四三条与 UI 文案一一对应

**预估**：2～3 人日

---

### M1-2 · CLI `qr ship-check` 【P0 · 已完成 2026-06-28】

**目标**：单命令聚合设计者最小验收（dev/qr 改完后的标准动作）。

**内容**：
```bash
qr ship-check              # 默认当前目录对应项目
qr ship-check -p dev/qr    # 指定项目
```
- 顺序：`qr doctor`（摘要）→ 项目相关 `unittest` 子集（若存在 tests/）→ 提示「Web 点验 URL」
- 退出码：doctor 有 error 级则非 0
- 输出末尾：建议 `qr log --type decision` 模板一行

**改动范围**：`qr/cli.py` 新命令；可选写入 `docs/USE_CASES.md`

**验收**：
- dev/qr 根目录执行通过且输出人类可读
- 失败时给出下一步建议（非堆栈）

**预估**：1 人日

---

### M1-3 · 「接着干」收束模式 【P1 · 已完成 2026-06-28】

**短板**：会推荐活跃项，不抑制多项目分散。

**内容**：
- 洞察/问答顶栏增加 **「本周主攻」**（读 `~/.qr/config.json` 或新键 `focus_project`）
- 非主攻项目 Cursor/活动仅折叠展示，不删数据
- 每周一 `qr summary` 时若未设主攻，提示选择 1 个

**验收**：
- 设置 `focus_project` 后「接着干」首卡为该项目的 Git/Cursor/README 待办
- 规范或 config 文档有说明

**预估**：2 人日

## 3. M2 · 引导语与决策闭环

> **短板**：761+ 对话 vs 个位数决策、引导语合并几乎未用；时间线自动涨、设计意图不沉淀。  
> **类型**：`闭环`

### M2-1 · 决策草稿自动生成 【P0 · 已完成 2026-06-28】

**目标**：里程碑时降低 `qr log decision` 门槛。

**内容**：
- 会话结束或手动触发：`qr decision draft` / Web「生成决策草稿」
- 输入：近 N 轮 Cursor 对话 + 相关 Git diff 摘要
- 输出：填充「问题 / 选项 / 结论 / 原因」模板，**不自动入库**，设计者改后保存
- 可选：长会话（>30 轮）每周提醒一次

**改动范围**：`qr/prompt_guides.py` 或新模块 `qr/decision_draft.py`；Web 决策区；MCP 可选 `qr_decision_draft`

**验收**：
- 对 dev/qr 最近一次长会话能生成可编辑 Markdown
- 保存后进入时间线且 `content LIKE '%决策记录%'`

**预估**：2～3 人日

---

### M2-2 · 引导语相似合并建议 【P0 · 已完成 2026-06-28】

**目标**：减少「收件箱永远不清」。

**内容**：
- `qr prompts suggest-merge`：对收件箱片段按 embedding 或字面相似聚类
- Web 引导语页：「建议合并」侧栏，勾选后一键 `merge`
- 规则：相似度 > 阈值且同 `project` 优先

**验收**：
- 对现有 12+ 片段能给出至少 1 组合理建议（人工确认）
- 合并后标记「合并合成」

**预估**：2 人日

---

### M2-3 · `执行-` / `参考-` 前缀提醒 【P1 · 已完成 2026-06-28】

**短板**：无前缀对话不进引导语，但用户常忘。

**内容**：
- Cursor 采集后：若标题无前缀且为 dev/qr 等注册项目，写入 `events` 同时记 `prompt_guides` 待办标记
- Web 时间线 Cursor 条目：轻提示「改为 执行- 主题 可进引导语」
- 洞察页计数：「本周无前缀执行类对话 N 条」

**验收**：
- 新 Cursor 事件无前缀时 Web 可见提示
- 改标题后 `qr prompts sync` 能进收件箱

**预估**：1～2 人日

## 4. M3 · Cursor 与工作区

> **短板**：`window` / unknown 工作区、混开 `~/QR` 导致统计与检索污染。  
> **类型**：`缺口`

### M3-1 · 工作区校验与告警 【P0 · 已完成 2026-06-28】

**内容**：
- `qr doctor` 增项：检测 Cursor 最近会话 `project` 是否为已注册 `~/QR/*/*` 路径

- Web 设置页：展示「当前 Cursor 应对齐的项目根」列表

**验收**：
- 本机历史 `window` 类条目 doctor 给出 warn（可配置忽略）
- 文档 `USE_CASES` 工作区一节链到本文

**预估**：1～2 人日

---

### M3-2 · 项目路径注册表强化 【P1 · 已完成 2026-06-28】

**内容**：
- `workspace` 模块维护 `cursor_roots`：uuid → 规范 `dev/qr` 映射
- 采集时 jsonl 路径无法解析则标 `needs_review`，不进错误 project 聚合
- `qr project` 体检显示「推荐 Cursor 打开路径」

**验收**：
- 新开 `~/QR/dev/qr` 会话 project 字段为 `dev/qr`
- 旧数据可通过一次性 `qr backfill` 或迁移脚本修正（可选）

**预估**：2～3 人日

### M3-3 · 长会话 checkpoint 【P1 · 已完成 2026-06-28】

**短板**：单日 300+ 轮无法复盘。

**内容**：
- 当单会话 `cursor:uuid` 事件 >40 轮：Web 时间线该会话显示「生成 checkpoint」
- checkpoint = 已完成 / 待办 / 风险 三段 Markdown，可写入 `~/.qr/notes/` 或 decision 草稿
- 规范建议：>50 轮新开 Chat（已在 STANDARDS，产品侧 reinforce）

**验收**：
- 对历史超长 uuid 手动触发可生成 checkpoint
- 纳入时间线可检索

**预估**：2 人日

### M4-1 · 扩展评测题集（叙事/口语） 【P1 · 已完成 2026-06-28】

**内容**：

- 题集分栏：`core`（9 题门禁）+ `extended`（不阻断发布，周报展示）

**验收**：

**预估**：1～2 人日

### M4-2 · 「仅出处」问答模式 【P1 · 已完成 2026-06-28】

**短板**：生成弱时用户难判断对错。

**内容**：

**验收**：

### M4-3 · 索引健康自动化 【P2 · 已完成 2026-06-28】

**短板**：doctor 报孤儿索引需人工 cleanup。

**内容**：
- `qr update` 结束可选 `--index-health`（config 默认每周一次）
- 自动 `index-health --cleanup` 仅删源不存在文档；写时间线
- 报告：清理数量 + 前 5 条路径

**验收**：
- 人为删除文件后下一轮 update 孤儿数下降
- 不误删仍存在的文件索引

**预估**：1 人日

---

### M4-4 · 检索阶段 D～F 【P3 · 维持】

**维持理由**：见 `RETRIEVAL_UPGRADE_PLAN` §2、§6。

**触发条件**（满足任一再立项）：
- `extended` 题集连续 2 月命中率 < 70%
- 用户反馈口语 miss 周均 > 5 次（可 Web 反馈按钮计数）
- 愿意接受检索延迟 +2～5s

**届时动作**：按 RETRIEVAL 文档阶段 D（多查询）优先，不直接上 cross-encoder。
