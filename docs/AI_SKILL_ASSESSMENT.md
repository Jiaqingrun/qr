# AI 使用水平评测规则

> 本文件为 **完整量表与对照框架**；全局规范摘要见 `standards/STANDARDS.md` §五。  
> 执行：`qr ai-assess --save`（日）· `qr eval monthly --save`（月）· 洞察页每日计划勾选。

## 1. 评测原则

1. **行为证据优先**：以 `~/.qr/qr.db`、屏幕采样、facts、eval 结果为主，不以自评问卷为主。
2. **多框架对照**：至少覆盖 QR 六维 + AISA 五维 + L 阶梯；重大评测可对照 AILit / PRL。
3. **纵向可比**：日快照 `~/.qr/assessments/`、月报 `~/.qr/eval_monthly/`；对照上月与 `docs/RETRIEVAL_UPGRADE_PLAN.md` 触发条件。
4. **不替代认证**：本评测供个人复盘与规划；对外证明须附 eval 快照与时间线。
5. **协作者角色边界**（与 `standards/STANDARDS.md` §四 一致）：本机默认 **设计者（本人）/ AI 实现（Cursor 等）/ 验收（本人）** 三角色。**禁止**用「是否会手写 Python」代替对设计能力、工具链或 AI 协作水平的评判。QR 由本人亲手设计、AI 协作为实现手臂，属正常路径。

## 1.5 评测时如何读「技术」

| 误区 | 正确读法 |
|------|----------|
| 看不懂源码 → 低档位 | 看 **决策笔记、规范版本、进化计划验收、doctor/eval 是否通过** |
| 代码行数多 → 一定是程序员 | 可能是 **AI 实现**；设计者看 **架构与验收** |
| L4 必须手写 MCP/RAG | L4 可以是 **设计并落地** qr 类系统 + 行为采集闭环 |
| AISA Technical 低 → 全面否定 | Technical 偏实现细节；**Workflow / Critical** 对设计者往往更重 |

**设计者高分证据**：模块划分合理、规范可执行、里程碑有决策、验收清单跑通。  
**常见扣分**（与会不会写 Python 无关）：对话多决策少、混开 workspace、裸「继续」、引导语未合并。

## 2. QR 六维工程化雷达（主量表）

各维 **1–10 分**，参照 **Personal AI Ops / 个人开发者** 群体；**10 ≈ 该维顶尖实践**。**不得**将六维综合分解释为「编程考级」。

| 维度 | 考察什么 | 高分证据（可观测） | 常见扣分 |
|------|----------|-------------------|----------|
| **提示工程** | 任务拆解、约束、迭代、可复用 prompt | 引导语精炼、前缀规范、多轮纠偏、验收标准明确 | 模糊目标（「继续优化」）、碎片未合并 |
| **工具链** | Personal AI Ops：采集/RAG/治理/评测/MCP | 自研或深度定制 qr 类系统、eval 达标、launchd | 仅聊天无归档、无评测 |
| **元认知** | 对 AI 能力边界与升级时机的判断 | 写清「不做什么」、触发条件、指标追问 | 无限堆功能、不读 eval 结果 |
| **复盘习惯** | log、总结、决策、评测节奏 | 日/月评测、决策笔记、周期 summary | 对话多决策少、无快照序列 |
| **多项目协作** | 工作区隔离、MCP project、PROJECT 分层 | 每仓独立 Cursor 根、facts 分项目 | 混开 workspace、跨项目检索污染 |
| **领域应用** | AI 在业务上的落地深度 | sports MVP、scribe 长篇（`qr log --type activity` 记非 Cursor 投入）、边缘部署等 | 只有规划无代码/无文档闭环 |

**综合分** = 六维算术均值（保留一位小数）。  
**档位**：≤6.0 进阶前 · 6.1–7.5 进阶 · 7.6–8.5 L4+ · 8.6–9.2 L4+ 逼近 L5 · ≥9.3 L5 单项触达。

## 3. 个人 AI 工程化 L1–L6

| 级别 | 特征 | 判定 |
|------|------|------|
| L1 | 偶尔问答 | 无规则、无归档 |
| L2 | Tab/Chat 熟练 | 少量 User Rules |
| L3 | Agent 多文件 | README/规范有，无 MCP/RAG |
| **L4** | MCP、RAG、规范闭环、行为采集 | **主档位参考** |
| L5 | 评测驱动、多项目可复制、稳定 SLO | 工具链或多项目单项触达 |
| L6 | 对外平台化、大规模协作 | 非个人当前目标 |

## 4. AISA 五维（对照用，估算 0–100）

权重：Prompting 23% · Critical 22% · Technical 20% · Workflow 25% · Safety 10%。

| 1–10 档 | 行为描述 |
|---------|----------|
| 1–2 Novice | 无意识地使用，无结构 |
| 3–4 Developing | 知道技能但 inconsistently |
| 5–6 Competent | 可重复的技巧 |
| 7–8 Proficient | 能解释为何这样做 |
| 9–10 Expert | 原则内化，重塑工作方式 |

**Proficient 参考线**：综合约 **75–89**；**Competent** 约 55–74。

## 8. 报告结构（完整版）

按需对话级评测（六维 + 多框架 + 90 天行动项）按下列结构输出；模板文件：`~/.qr/templates/ai_assess_full.md`（首次 `qr ai-assess --save` 时从仓库复制）。

1. **执行摘要**（等级、综合分、最强/最弱项、相对上次 Δ）  
2. **协作者角色 · 分项证据表**（设计者 / AI 实现 / 验收）  
3. **硬数据画像**（`qr ai-assess`、`qr eval rag`、月报摘要）  
4. **六维分项 + 证据**  
5. **多框架对照表**（AISA、L、AILit/PRL）  
6. **优势 / 短板 / 90 天建议**  
7. **附录**：数据时间、`qr doctor`、检索 core/extended  

### 8.1 协作者角色证据表示例

| 角色 | 承担者 | 可观测证据（本周期） | 读法 |
|------|--------|----------------------|------|
| **设计者** | 本人 | 决策笔记、规范版本、进化计划验收、`ship-check`、模块/约束清晰度 | 高分 ≠ 会写 Python；看架构与验收是否跑通 |
| **AI 实现** | Cursor 等 | Git diff、unittest、多文件落地记录 | 实现债多不直接扣设计者档 |
| **验收** | 本人 | `qr doctor`、core 9/9、`qr ship-check`、Web/CLI 点验 | 缺验收记录应在「复盘习惯」维体现 |

**反面示例（禁止）**：「看不懂 `query.py` 源码 → 工具链 4 分」。  
**正面示例**：「近 30 天 Cursor 对话 400+、决策笔记 2 条 → 复盘习惯扣分；工具链仍可为 8（设计并落地 qr）」。

## 9. 与检索升级的关系

内置检索 **连续 2 月低于 9/9** 或延迟明显恶化时，对照 `docs/RETRIEVAL_UPGRADE_PLAN.md` 考虑阶段 C+；否则优先体验、资产化与多项目落地。
